# Workspace rules

This file lists, in plain language, what the AI assistant (Copilot) is
allowed and not allowed to do inside this workspace. It complements the
machine-readable `.github/copilot-instructions.md`.

## What Copilot may write

Copilot may **only** create or edit files under `review/`:

- `review/findings.md` — the human-readable review.
- `review/actions.json` — the declarative action plan.
- `review/scripts/*.py` — Python scripts that use only `kibridge_api`.

## What Copilot may not modify

Copilot must **never** edit:

- `snapshot/*` — regenerated by the KiBridge plugin on every "Open Workspace". Edits here are wasted; they get overwritten.
- The `.kicad_pcb`, `.kicad_sch`, or `.kicad_pro` files. Only the plugin writes those.
- Any file outside `kibridge_workspace/`, including the plugin source, the router source, install scripts, or this `RULES.md`.
- `.github/copilot-instructions.md` (unless explicitly asked).

If Copilot thinks one of those files needs to change, it should write a *recommendation* in `findings.md` — never edit directly.

## What Copilot must not invent

- **Component values.** Resistor values, capacitor values, voltage ratings — if not in `footprints.json`, write `(not in snapshot)`.
- **Numeric estimates.** Currents, voltage drops, trace temperatures — if you're computing or guessing, label as *Inferred* with the assumption listed.
- **Reference designators.** Discuss only refdes that appear literally in `footprints.json`. Don't invent R47, C12, etc. to fill out a list.
- **Component types not present.** Don't claim there's a MAX485 or an Arduino if no footprint with that value exists.

## Required structure for `findings.md`

Use three sections, in this order:

```markdown
## Verified
(Facts read directly from snapshot/. Each line cites the source file.)

## Inferred
(Reasonable conclusions drawn from snapshot data. Each line states the
assumption and lists the question the user should confirm.)

## Unknown
(Things that would matter but are not present in snapshot/. Each line
says what's missing and why it matters.)
```

If you would have written a section that doesn't fit those three, you're guessing without flagging it. Move it to *Inferred* and add a question.

## Action plan rules

- Use only ops from the whitelist in `.github/copilot-instructions.md`.
- Modifying ops require `"confirm_changes": true` at the top of `actions.json` — otherwise the plugin rejects the whole file.
- Coordinates must be inside the board outline (check against `tracks.json` extents).
- Net names must exist in `board_inspect.json`.

## Keep it small

If the issue can be addressed in 3 actions, don't write 30. The human reads every entry.
